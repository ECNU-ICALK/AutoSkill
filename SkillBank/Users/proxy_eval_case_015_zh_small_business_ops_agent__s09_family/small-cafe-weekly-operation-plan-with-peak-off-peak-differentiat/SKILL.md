# small-cafe-weekly-operation-plan-with-peak-off-peak-differentiation-and-emergency-staffing

为小型独立咖啡店（1–3人）设计的每周运营计划技能，既严格区分工作日与周末的客流特征以优化排班、备货、促销和损耗控制，又内嵌突发缺人场景下的即时响应机制，确保常态高效与应急韧性双重保障。

## Prompt

# Goal
生成一份结构化、可执行的咖啡小店每周运营计划，严格区分工作日（周一至周五）与周末（周六、周日）两套策略，覆盖排班、备货、促销、损耗控制四大模块；同时内嵌突发缺人应急协议，支持单人/双人在岗状态下的岗位熔断、角色复用与外部补位，所有内容基于小规模、高周转、重体验的真实场景，拒绝通用模板，所有动作需具象到时间、角色、数量、判断条件、防错提示及服务底线红线。

# Constraints & Style
- 必须显式分栏呈现：【工作日策略】vs【周末策略】，不可混述；
- 排班：明确标注每时段在岗人数、岗位职责、轮休机制；突发缺勤时必须同步激活应急协议——区分「单人坚守」与「双人协作」状态，按预设优先级熔断非核心任务（如社群发图、豆单更新），释放人力至顾客接触点，并明确角色复用规则（如出品岗代收银但须保‘3秒响应’原则）；应急联络方式须为可直接点击/扫码形态（如预填拨号链接、微信二维码链接）；
- 备货：所有计算必须含动态公式（如：日均杯量 × 单杯用量 × 天数 × 安全系数），并标注安全系数取值逻辑（如鲜奶用1.05因易腐，耗材用1.2因损耗波动大）；
- 促销：工作日侧重引流与私域沉淀（如早鸟入群），周末侧重转化与峰值体验（如动线观察、彩蛋核销），禁用无差别折扣；
- 损耗控制：工作日聚焦流程优化与微复盘（如晨会1分钟废弃分析），周末聚焦高峰承压与实时响应（如超3分钟等待即赠饮）；
- 应急协议所有动作必须满足：≤2个操作步骤、无需额外培训、不依赖专用设备（仅手机+吧台即可执行），禁止模糊表述（如“灵活调整”），全部替换为具体动作+触发条件（例：“当B岗未到且>8:00，A岗启动‘快取模式’：关闭手冲区，只接小程序预订单+基础意式”）；
- 禁止出现任何未脱敏实体：不得包含具体城市、店名、人名、供应商名称、APP完整名（可用「接龙管家」代称）、微信/二维码等联系方式；
- 所有建议工具须为通用型免费/轻量级（如「Tide」APP、「企业微信」后台），不绑定特定SaaS系统；
- 语言简洁，使用符号（✅⚠️🔴💡）提升可扫视性，避免段落式描述；应急部分禁用emoji，保持纯文本分块结构以利快速扫描执行；
- 应急响应必须在确认缺勤后30分钟内可启动，仅依赖现有角色、物理资产（如打印SOP卡、小白板、杯 sleeve）及预审社区资源（如<NEIGHBOR_BUSINESS>、<STUDENT_POOL>、<MEMBER_NAME>），严禁新增培训、招聘或专用工具；
- 严格禁止危及食品安全、设备安全或顾客信任的行为（如跳过洗手、提供未验证新品、虚假岗位宣称）；
- 应急逻辑采用三级响应：Level 1（内部角色合并）、Level 2（流程折叠）、Level 3（外部激活），每级含明确触发条件与硬性终止规则；
- 输出必须包含强制性事后闭环：根因记录、联系人列表更新、个性化客户善意补偿（如手写感谢卡+小样）；
- 所有应急动作须附带服务底线红线（如“收银可由出品岗代做，但‘3秒响应’原则不可降级”）。

# Workflow
1. 解析用户提供的日均客流区间（如“30–80人”）或隐含峰谷比（如“工作日和周末客流差异大”），据此设定工作日基准量与周末放大系数（默认×1.8，可依用户反馈调整）；
2. 分别构建工作日与周末的四模块策略表，每表字段统一为：模块名｜策略名｜执行动作（≤3步）｜关键逻辑/风控点｜成效验证方式；
3. 在各模块末尾附加「执行口诀」或「铁律」（如“鲜奶按天订，豆子按周订”），强化记忆与落地；
4. 对排班模块，自动注入应急协议分支：根据在岗人数（1人／2人）及距营业开始时间（＞30min／≤30min）触发对应熔断-复用-补位流程；
5. 输出前校验：所有公式可反向推算、所有动作有明确责任主体（A岗/B岗/全员）、所有预警红线具可操作性（如“燕麦奶＜3箱→菜单加注”）、所有应急动作含明确触发条件与服务底线（如“收银可由出品岗代做，但‘3秒响应’原则不可降级”）；
6. 应急响应执行后，强制追加三项闭环动作：① 根因归档（缺勤类型+时间+影响节点）；② 更新预审资源联系人列表；③ 向受影响顾客发送个性化善意补偿（形式限于店内可执行项，如手写卡+小样）。
