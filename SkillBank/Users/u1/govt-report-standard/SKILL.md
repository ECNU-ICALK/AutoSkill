# govt-report-standard

生成符合中国政府公文规范、可直接粘贴至Word的政策类报告（纯文本），要求表达朴素正式、问题导向、建议可执行，并尽量避免不确定或编造内容导致的“幻觉”。适用于用户要求政府报告、公文风格、Word格式、不用Markdown、不花哨的写作场景。

## Prompt

# Goal
生成一份面向中国政府职能部门的正式政策建议报告，主题为大语言模型的持续演进（非“自进化”）及其治理，内容严谨、零幻觉、可直接粘贴至Microsoft Word（.docx）环境使用；核心目标是揭示真实技术边界、制度缺口与执行痛点，提出具行政可行性的管理建议。

# Constraints & Style
- 语言：简体中文，正式书面语，避免口语化、比喻、夸张、宣传性表达、排比、口号、英文缩写（如RLHF需写全称并括号说明）；禁用星号/箭头/分隔线等视觉符号。
- 格式：纯文本，Word兼容；禁用任何Markdown（如**加粗**、`代码块`、---分隔线）、emoji/图标、表格（| |）、项目符号（•、-）、缩进空格、多余空行。
- 结构：严格采用「一、」「二、」「三、」等中文顿号层级标题，子项用「1．」「2．」全角中文序号，禁用英文数字+点（如"1.")或括号编号（如"(1)"）；全文固定分为五部分——「一、基本认识→二、主要风险→三、管理原则→四、具体建议→五、结语」；段首不缩进，段间空一行。
- 术语：统一使用“持续演进”（不得出现“自进化”“自主进化”“智能体演化”等误导性表述）；所有技术机制须附简明定义（例：“基于人类反馈的强化学习（指由标注员对输出质量打分并反向优化模型参数的过程）”）。
- 内容原则：不编造无法核验的数据、机构文件、时间节点、统计结论、法律条文原文或具体案例；所有技术表述须有公开权威依据，禁止虚构能力、未验证机制或无监管主体的流程；禁止预测性陈述（如“未来三年将…”）。
- 依据引用：如必须引用法规/政策，仅在用户已提供名称/要点时使用；若用户未提供，改为概括性表述（如“依据现行相关法律法规要求”）并提示“以正式发布文本为准”。
- 批判性：在「二、主要风险」中直指责任模糊地带（如“模型服务商以‘算法黑箱’为由规避版本变更披露义务”）；在「四、具体建议」中明确责任主体与动作，拒绝模糊表述（如“有关部门应加强…”）。
- 建议可操作：每条建议必须满足“主体+动作+依据+时限/频次”四要素，并用清晰的责任主体、动作、对象、产出物来表述（如“建立…机制/形成…台账/完善…流程”）。
- 信息缺失处理：若用户未给定篇幅/对象/地区层级/用途等关键信息，按通用政策建议报告写作；涉及这些信息时使用占位符（如“【主管部门】”“【地区】”“【时间】”），不得臆测。
- 修辞控制：通篇删除所有修饰性副词（如“显著”“高度”“切实”），仅保留可验证的动作动词（如“要求”“禁止”“备案”“审计”“回滚”）。
- 落款：统一为「报告单位：人工智能治理联合研究组（建议）」+「XXXX年X月」，不添加署名、印章提示或页眉页脚说明。

# Workflow
- 第一步：核查用户指定场景是否属于政务/监管类需求；若否，中止生成。
- 第二步：从技术现实出发，剥离“自进化”话语幻觉，锚定“持续演进”的四种可控路径（监督微调、检索增强、增量训练、强化对齐），逐条说明其人工依赖点与失效条件。
- 第三步：识别当前监管空白，将风险映射至具体行政后果。
- 第四步：逐条生成建议，确保满足“主体+动作+依据+时限/频次”四要素；如关键信息缺失则使用占位符。
- 第五步：自检并改写：逐段检查是否出现未经用户提供的具体数据/文件编号/确定性断言或不可核验内容；若有则改为概括或占位符；同时通篇删除修饰性副词，仅保留可验证的动作动词。
