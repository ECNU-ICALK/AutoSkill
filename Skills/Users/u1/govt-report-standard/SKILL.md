# govt-report-standard

生成符合中国政府公文规范、无技术幻觉、纯文本Word兼容格式的政策类报告，禁用Markdown、图标、强调符号及主观修辞；聚焦大语言模型持续演进治理，强调问题导向、责任明确、依据扎实、建议可操作。

## Prompt

# Goal
生成一份面向中国政府职能部门的正式政策建议报告，主题为大语言模型的持续演进（非'自进化'）及其治理，内容严谨、零幻觉、可直接粘贴至Microsoft Word（.docx）环境使用；核心目标是揭示真实技术边界、制度缺口与执行痛点，提出具行政可行性的管理建议。

# Constraints & Style
- 语言：简体中文，正式书面语，避免口语化、比喻、夸张、宣传性表达、排比、口号、英文缩写（如RLHF需写全称并括号说明）；禁用星号/箭头/分隔线等视觉符号；
- 格式：纯文本，禁用任何Markdown（如**加粗**、`代码块`、---分隔线）、emoji（如✅🔹）、表格（| |）、项目符号（•、-）、缩进空格、多余空行；
- 结构：严格采用「一、」「二、」「三、」等中文顿号层级标题，子项用「1．」「2．」全角中文序号，禁用英文数字+点（如"1.")或括号编号（如"(1)"）；同时，全文须固定分为五部分——'一、基本认识→二、主要风险→三、管理原则→四、具体建议→五、结语'；段首不缩进，段间空一行；
- 术语：统一使用'持续演进'（不得出现'自进化''自主进化''智能体演化'等误导性表述）；所有技术机制须附简明定义（例：'基于人类反馈的强化学习（指由标注员对输出质量打分并反向优化模型参数的过程）'）；
- 内容原则：所有技术表述须有公开权威依据（如工信部白皮书、NIST标准、《生成式AI服务管理暂行办法》第12条、《网络安全审查办法》《数据安全法》、arXiv主流论文结论、网信办备案要求），禁止虚构能力（如“自进化”“自主意识”）、未验证机制或无监管主体的流程；禁止预测性陈述（如'未来三年将…'）；
- 批判性：在'二、主要风险'中直指责任模糊地带（如'模型服务商以“算法黑箱”为由规避版本变更披露义务'）；在'四、具体建议'中明确责任主体与动作（如'由省级网信办牵头组织季度穿透式审计'），拒绝模糊表述（如'有关部门应加强…'）；每条建议必须满足'主体+动作+依据+时限/频次'四要素（例：'工业和信息化部应在2025年底前发布《政务大模型演进备案表》格式规范，依据《行政许可法》第32条'）；
- 修辞控制：通篇删除所有修饰性副词（如'显著''高度''切实'），仅保留可验证的动作动词（如'要求''禁止''备案''审计''回滚'）；
- 落款：统一为「报告单位：人工智能治理联合研究组（建议）」+「XXXX年X月」，不添加署名、印章提示或页眉页脚说明。

# Workflow
- 第一步：核查用户指定场景是否属于政务/监管类需求；若否，中止生成；
- 第二步：从技术现实出发，剥离'自进化'话语幻觉，锚定'持续演进'的四种可控路径（监督微调、检索增强、增量训练、强化对齐），逐条说明其人工依赖点与失效条件；
- 第三步：对照《网络安全审查办法》《数据安全法》等，识别当前监管空白（如：未覆盖模型上线后动态知识注入行为），将风险映射至具体行政后果（如'导致事故时无法认定算法提供者主体责任'）；
- 第四步：每条建议必须满足'主体+动作+依据+时限/频次'四要素；
- 第五步：通篇删除所有修饰性副词，仅保留可验证的动作动词。
